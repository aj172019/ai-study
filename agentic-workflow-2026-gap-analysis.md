## 2. 세션별 핵심 요약

### Session 1 — 하네스 엔지니어링 (카카오 황민호)

- **하네스 3요소**: `CLAUDE.md` (프로젝트 맥락) + `.claude/skills/` (HOW) + `.claude/agents/` (WHO)
- **6가지 오케스트레이션 패턴**: Pipeline / Fan-out-Fan-in / Expert Pool / Producer-Reviewer / Supervisor / Hierarchical Delegation
- **WHO/HOW 분리 원칙**: 에이전트는 페르소나·책임·도구, 스킬은 절차·출력 형식
- **정량 효과**: +60% 품질 향상, -32% 분산(std. dev.) — 카카오 내부 측정
- **메타하네스 개념**: 하네스 자체를 개선하는 하네스 (자가 평가·개선 루프)

### Session 2 — 자율형 AI 협업 (카카오 황민호)

- **description이 라우팅 핵심**: 에이전트·스킬의 description이 모호하면 잘못된 것이 호출됨
- **Handoff 계약**: Analyzer→Developer에 파일 경로 + 패턴 체크리스트 + Envelope 형식 고정
- **역할 오염(Role contamination) 방지**: 한 에이전트가 여러 역할을 겸직하면 페르소나가 흐려짐
- **"왜(intent)"를 명시**: description에 "왜 이 스킬/에이전트가 필요한가"를 적어야 라우팅 정확도↑

### Session 3 — 당근 베스트 프랙티스 (이예찬)

- **관측 가능성(Observability)이 출발점**: 에이전트가 뭘 하는지 raw I/O를 보지 않으면 개선 불가
- **사용자별 권한 기반 도구 필터링**: LLM에 모든 도구를 주지 말고, 사용자 컨텍스트에 맞게 **프론트에서 먼저 필터링**
- **카비(Karbi) 파이프라인 6단계**: 메시지 수신 → 중단 감지 → 사용자 정보 조회 → LLM 루프 → 도구 실행 → 응답
- **최소 컨텍스트 원칙**: 에이전트에 주는 컨텍스트는 작업에 꼭 필요한 것만 (토큰 비용 + 혼란 감소)
- **3대 한계**: ①자가 피드백 루프 부재 ②토큰 비용 관리 어려움 ③실패 패턴 자동 발굴 부재

### Session 4 — AI 평가 프레임워크 (스페이스와이 황현태)

- **엔터프라이즈 AX의 진짜 문제**: 암묵지의 형식지화 — 기술 문제가 아니라 조직 지식 문제
- **평가의 본질 = "재현의 정의"**: "무엇을 '맞췄다'로 할 것인가"부터 합의해야 평가가 가능
- **4가지 재현 유형** (세션 4 원문 기준, 라인 365~370):
  - **의미 재현 (Semantic)**: LLM-as-judge / 임베딩 유사도 / rubric → 요약·분석·기획 문서
  - **구조 재현 (Structural)**: JSON Schema / Pydantic / 파싱 → API 응답·보고서·리스트
  - **형식 재현 (Literal)**: exact string match → 코드·SQL·설정 파일
  - **행동 재현 (Behavioral)**: 실행 후 결과 비교 → 코드·SQL·변환 작업
- **엔터프라이즈 AX 4대 요소**: 컨텍스트 + 평가 + 레거시 통합 + 리더십
- **폐쇄망 반전**: 폐쇄망은 AX 장벽이 아니라 fail-safe. 외부 유출 채널이 없으면 에이전트 사고가 외부로 못 나감 → **제약이 안전을 만든다**
- **AX 팀 유형 4가지**: v1 중앙 FDE / v2 바이브코더 교육 / v3 Beachhead(단일 부서 타격) / v4 꼭짓점 AX(리더 변환)
- **고성과자 의존의 역설**: PoC는 고성과자 한 명이 끌지만, 전사 확산은 평균 구성원으로도 돌아가야 함

### Session 5 — AI 에이전트와 신뢰 (위데이터랩 강승우)

- **신뢰성 향상의 처방 = 절차(Procedure) + 툴 지식(Tool Knowledge)** (2축 모델)
- **절차(시간 축)**: "1단계: 입력 검증 → 2단계: 권한 확인 → 3단계: 도구 호출 → 4단계: 결과 검증" — 디버깅·재현·감사·학습 4가지를 동시에 만족
- **툴 지식(공간 축)**: 도구의 용도 + 사용 조건 + 실패 시 대안 + 비용. 단순 description으로는 부족 — **"언제·왜·금지·대안"까지 명시**해야 신뢰성 확보
- **잔고장(Petty Failure)** 이 신뢰 붕괴의 주범: API 오류·도구 응답 형식 변경·외부 의존성 깨짐·모델 포맷 어긋남
- **비-Episodic 입력**: 사용자는 "그거 부탁해"라고 부족하게 명령 → 에이전트가 목적/제약/출력 형식 슬롯을 명시적으로 채워달라고 되묻는 의무
- **명령 샤딩(Command Sharding)**: 한 기능이 여러 에이전트·세션으로 흩어지면서 master intent가 사라지는 현상

## 6. Phase 로드맵

### 🔨 Phase 1 (본 리포트 작성 시점 — 실제 하네스 도입 중)

## 7. 부록 — 세션 4 "재현의 정의" 원문 매트릭스

> 세션 4 황현태(스페이스와이) 발표 라인 365~370. C1 `/evaluate-ai-output` 스킬의 직접적 source.

| 재현 유형 | 정의 | 평가 방법 | 적합한 태스크 |
|----------|------|----------|--------------|
| **의미 재현** (Semantic) | 핵심 정보·결론이 일치 | LLM-as-judge / 임베딩 유사도 / rubric 기반 채점 | 요약·분석·기획 문서 |
| **구조 재현** (Structural) | 출력 포맷·필드·순서가 일치 | 스키마 검증 (JSON Schema / Pydantic) / 파싱 | API 응답·보고서·리스트 |
| **형식 재현** (Literal) | 토큰 단위까지 동일 | exact string match | 코드·SQL·설정 파일 |
| **행동 재현** (Behavioral) | 같은 결과를 내는 다른 표현 | 실행 후 결과 비교 (코드 실행, DB 쿼리 결과) | 코드·SQL·변환 작업 |

### 핵심 통찰

- **태스크마다 재현 정의가 다르다**: SQL은 형식 재현 불필요(다른 문법이라도 결과 같으면 OK → 행동 재현). 정중한 고객 응대 메일은 같은 정보라도 톤이 다르면 실패 → 스타일 재현까지 필요.
- **LLM-as-judge의 순환 문제**: 의미 재현을 평가하는 judge 자체가 비결정론적 → 평가가 재현되지 않음. 해결: 앙상블 채점, 인간 라벨 상관도 검증, rubric 구체화.
- **관측 가능성과의 연결** (S3): raw 로그가 있어야 형식/구조 재현이 가능. 의미 재현은 raw 로그만으로는 부족, rubric·인간 라벨이 별도 필요.
